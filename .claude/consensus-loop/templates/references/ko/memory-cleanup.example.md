# 메모리 정리 기준

> 회고 시 메모리 파일을 감사하는 기준입니다. 프로젝트에 맞게 조정하세요.

## 삭제 대상

- **코드에서 파생 가능** — `grep`/`Read`로 현재 코드에서 얻을 수 있는 정보 (API 타입 정의, SDK 레퍼런스 등)
- **git에서 파생 가능** — `git log`/`git blame`으로 얻을 수 있는 정보 (변경 이력, 누가 뭘 바꿨는지)
- **CLAUDE.md에 이미 존재** — 중복 기록
- **stale 프로젝트 상태** — "다음 세션 작업", "진행 중" 등 시간이 지나 무효해진 항목
- **완료된 에피소드** — 해결된 버그/트랙의 맥락 (원칙은 유지, 맥락은 삭제)

## 유지 대상

- **실패에서 나온 원칙** — "왜 실패했고 다음에 어떻게 방지하는가"
- **사용자 선호/피드백** — 협업 방식, 코드 스타일, 소통 패턴
- **프로세스 규칙** — TDD 사이클, 감사 워크플로우, HITL 기준
- **활성 프로젝트 맥락** — 현재 진행 중인 트랙의 상태

## 정리 절차

1. 메모리 파일 전체 목록 확인 (`Glob`)
2. 각 파일 Read → 삭제/유지/수정 판단
3. stale 섹션 트림 (파일 전체가 stale이면 삭제)
4. MEMORY.md 인덱스 정합성 확인
5. MEMORY.md 200줄 이내 유지

## 주의

- **삭제 전 사용자 확인** — 활성 트랙 여부를 모르면 물어보기
- 도구 출력(Knip, lint 등)은 가설 — 무비판적 삭제 금지
